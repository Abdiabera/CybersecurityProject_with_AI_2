Application Layer — Where Users Meet the Network
The Application Layer in Context
📐
Where It Sits

The Application Layer is Layer 7 in the OSI model — the top layer. In the TCP/IP model, it combines the OSI Application, Presentation, and Session layers into one. It is the layer where users actually interact with the network — every service a user touches operates here.

Client Role

The Client Initiates

A client is any device that sends a request to a server. Browsers, mail clients, FTP clients, and SSH terminals are all client software. A single device can run multiple client types simultaneously.

Server Role

The Server Responds

A server is a host running software that provides services to clients. Web servers, email servers, DNS servers, and file servers all listen on defined port numbers and respond only when a client requests something.

URI

Uniform Resource Identifier

The broadest term. A URI is any string that identifies a specific network resource. Both URN and URL are types of URI. The entire address is the URI.

URN

Uniform Resource Name

Identifies the name of a resource — the hostname and path. Does NOT include the protocol. Example: a book ISBN or DOI for an academic paper.

URL

Uniform Resource Locator

Defines the network location of a resource. Includes the protocol (HTTP, HTTPS, FTP, SSH). Used by browsers every time you navigate to a site.

Fragment

Section Pointer

The # symbol followed by a label. Points to a specific section within a page. Example: #page155 jumps directly to that section without loading a new page.

Memory tip: URI is the umbrella. URL adds the protocol (how to get there). URN names what it is without specifying how to get there. Every URL is a URI. Not every URI is a URL.
Module 16 — Topics 2 through 7
Application Layer Protocols
🌐
DNS — Domain Name System  Port 53

DNS translates human-readable domain names (like www.cisco.com) into IP addresses that routers can use. Without DNS, users would have to type the IP address of every website they visit.

1
Host Requests

A host types a domain name — the DNS client sends a query to the DNS server.

2
DNS Server Checks

The DNS server looks up the domain name in its records table.

3
Queries Up the Hierarchy

If the local DNS server has no record, it queries another DNS server higher up.

4
IP Address Returned

The correct IP address is sent back to the requesting host, which then contacts the server directly.

🌍
HTTP & HTTPS — Web Protocols  Port 80 (HTTP)  Port 443 (HTTPS)
HTTP — Hypertext Transfer Protocol
  • Used by browsers to request web pages from web servers
  • Server listens on Port 80
  • Data transmitted in plain text — not encrypted
  • Can be intercepted and read by anyone on the network
  • HTML defines how the returned page is displayed
HTTPS — HTTP Secure
  • Encrypts all data between browser and server
  • Server listens on Port 443
  • Browser displays HTTPS and a padlock icon
  • Required for any site handling sensitive user data
  • Interception yields unreadable ciphertext
Security Exam Fact: HTTP (Port 80) = plain text = risk. HTTPS (Port 443) = encrypted = required standard. If you see HTTP traffic carrying credentials in a network capture, that is a finding.
📁
FTP — File Transfer Protocol  Port 21 (Control)  Port 20 (Data)

FTP is used specifically for transferring files between devices across a network. It is unique because it uses two separate ports simultaneously — one for commands, one for data.

1
Control Connection — Port 21

Client opens a control connection to the server on port 21. Login credentials and file management commands (upload, download, rename, delete) travel through this connection. Stays open for the full session.

2
Data Connection — Port 20

A separate data connection is opened on port 20 when a file transfer begins. The actual file content travels here. Closes when the transfer completes.

Security Note: FTP sends everything in plain text — including usernames and passwords. The secure replacement is SFTP (Secure File Transfer Protocol), which encrypts the entire session using SSH. Standard FTP is treated as a vulnerability in any modern security policy.
💻
Telnet & SSH — Remote Access  Port 23 (Telnet)  Port 22 (SSH)
Telnet — Virtual Terminal
  • Introduced in the early 1970s — one of the oldest protocols
  • Creates a vty (virtual terminal) session to a remote device
  • Gives access to the remote device's CLI
  • All data — including passwords — sent in plain text
  • Server listens on TCP Port 23
  • Still used in lab/learning environments
SSH — Secure Shell
  • Direct replacement for Telnet
  • Same CLI access — but entire session is encrypted
  • Supports stronger authentication including key-based auth
  • Server listens on TCP Port 22
  • Required standard in all modern network environments
  • Intercepted SSH traffic = unreadable ciphertext
Best Practice: Always use SSH instead of Telnet. Telnet in production is a finding. The only variable that changes is the protocol — the level of access is the same. The difference is whether an interceptor gets valid credentials or meaningless ciphertext.
✉️
Email Protocols — SMTP, POP3, IMAP4
ProtocolFull NamePortRoleKey Behavior
SMTP Simple Mail Transfer Protocol 25 Sending email Pushes email from client to server and between servers. Handles all outbound mail movement.
POP3 Post Office Protocol v3 110 Retrieving email Downloads email to local device. Typically removes messages from server after download. Single-device access.
IMAP4 Internet Message Access Protocol v4 143 Retrieving & managing email Keeps messages on the server. Accessible from multiple devices simultaneously. Changes sync across all devices.
Remember: SMTP sends. POP3 downloads and removes. IMAP4 syncs and retains. IMAP4 is the modern standard for multi-device access.
📞
VoIP — Voice over IP

VoIP converts analog voice signals into digital data and encapsulates them into IP packets for transmission over the network. Uses peer-to-peer technology similar to instant messaging. A Call Manager server sits at the center, handling call setup, routing, and teardown between voice clients. To call traditional phone numbers, a PSTN gateway (Public Switched Telephone Network) is required. Specific protocols and ports vary by software — common protocols include SIP and RTP.

Module 17 — Troubleshooting Commands
Five Essential Network CLI Tools

ipconfig

C:\> ipconfig

Displays the IP configuration of a Windows device — IP address, subnet mask, and default gateway. The first command in any troubleshooting sequence.

ping

C:\> ping 10.10.10.1

Tests connectivity by sending ICMP Echo Requests to a target. If a reply is received, connectivity is confirmed. If it times out, the path or host has a problem.

netstat

C:\> netstat

Displays all active network connections, listening ports, and protocol status. Used to identify established sessions and detect unexpected or unauthorized connections.

tracert

C:\> tracert cisco.com

Traces the full route that packets take from the local device to a destination, showing every hop and the time taken at each one. Identifies where on the path a failure occurs.

nslookup

C:\> nslookup cisco.com

Queries a DNS server to resolve a domain name to an IP address. Used to diagnose DNS-related failures — if ping by IP works but by hostname fails, use nslookup.

Interpreting Ping Results
What the Ping Output Tells You
Ping Succeeds

Connectivity Confirmed

Packets Sent = Received, Lost = 0. The path between the two devices is functional. Response time in milliseconds shows network latency.

Request Timed Out

No Reply Received

The echo request was sent but no reply came back. The destination is unreachable, offline, or a firewall is blocking ICMP packets along the path.

IP Works, Hostname Fails

DNS Problem

If ping 10.10.10.1 succeeds but ping cisco.com fails — network connectivity is fine. The issue is specifically DNS resolution. Use nslookup to confirm.

Ping Gateway First

Narrow the Scope

Always ping the default gateway before testing external addresses. If the gateway responds — the problem is beyond the local network. If it fails — the problem is local.

# Step 1 — Check local configuration ipconfig /all # Step 2 — Test local connectivity (ping the gateway) ping 10.10.10.1 # Step 3 — Test internet connectivity (ping by IP) ping 8.8.8.8 # Step 4 — Test DNS resolution (ping by hostname) ping www.cisco.com # Step 5 — If Step 4 fails, query DNS directly nslookup www.cisco.com # Step 6 — Trace the path to identify where traffic is failing tracert www.cisco.com
Troubleshooting Workflow: Work from local to remote. Confirm the device has a valid IP address first. Then test the gateway. Then test the internet. Then test DNS. Each step confirms a layer of connectivity and narrows the source of the problem.
