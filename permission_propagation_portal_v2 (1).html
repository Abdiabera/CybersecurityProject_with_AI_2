Scenario 01 / 04
Moving Data Within the Same Volume
When you move a file within the same drive or partition, the file system only updates its directory pointer — the file itself is never recreated. Because nothing is rewritten, the original permission bits remain exactly unchanged, regardless of what permissions the destination folder has.
🏠
Think of it like moving furniture between rooms in the same house. The furniture does not change — only its location does. The rules attached to it stay exactly the same.

$ chmod u=rw,g=r,o=r HR/Report.docx
# Move within same volume
$ mv HR/Report.docx Finance/
# Verify permissions
$ ls -l Finance/Report.docx
-rw-r--r-- 1 solomon solomon 0 May 4 09:00 Finance/Report.docx
✓ RESULT: Permissions preserved — rw-r--r-- unchanged
Terminal — Try It Yourself
# Step 1: Create directories
$ mkdir -p HR Finance
# Step 2: Create file and set permissions
$ touch HR/Report.docx
$ chmod u=rw,g=r,o=r HR/Report.docx
# Step 3: Move within same volume
$ mv HR/Report.docx Finance/
# Step 4: Verify — expect rw-r--r-- (unchanged)
$ ls -l Finance/Report.docx
-rw-r--r-- 1 solomon solomon 0 May 4 09:00 Finance/Report.docx
💡
Key Insight: On Linux, moving within the same volume only updates the directory entry (inode pointer). No new file object is created. Since the file itself is untouched, its permission bits remain exactly as set — the destination folder's permissions have zero influence.

Scenario 02 / 04
Copying Data Within the Same Volume
Copying creates a brand-new file object. On Linux, that new file's permissions are derived from the original file's permissions filtered through the system's umask (commonly 022) — not from the destination folder. On Windows, the copy inherits the destination folder's permissions.
📋
Copying is like photocopying a document. The photocopy is a brand new sheet of paper. On Linux it starts with your original rulebook filtered through a policy filter (umask). On Windows it starts fresh with the destination building's rules.
⊞ Windows NTFS
Copying within the same volume creates a new file that inherits permissions from the destination folder. The copy gets Finance folder's ACL rules applied automatically.
🐧 Linux
Copying creates a new file whose permissions are derived from the original file's permissions filtered by the system umask. The destination folder permissions have no direct effect.

📁

bash — solomon@PSBOS-SSeifu
$ touch HR/Report_copy.docx
$ cp HR/Report_copy.docx Finance/
$ ls -l Finance/Report_copy.docx

⚠ RESULT: New file — permissions from umask, NOT Finance folder

# To preserve original permissions exactly:
$ cp -p HR/Report_copy.docx Finance/
Terminal — Try It Yourself
# Create file in HR
$ touch HR/Report_copy.docx
# Copy to Finance (same volume)
$ cp HR/Report_copy.docx Finance/
# Verify — umask applied, not Finance folder permissions
$ ls -l Finance/Report_copy.docx
-rw-r--r-- 1 solomon solomon 0 May 4 09:00 Finance/Report_copy.docx
# Preserve original permissions exactly:
$ cp -p HR/Report_copy.docx Finance/
💡
Key Insight: cp without flags creates a new file using the source's permissions filtered by the system umask (typically 022, which removes group-write and other-write bits). Use cp -p to fully preserve original permissions. This is different from Windows where the copy inherits the destination folder's ACL.

Scenario 03 / 04
Moving Data to a Different Volume
Moving across volumes behaves very differently between Windows and Linux. On Windows, cross-volume move = copy + delete, so the file inherits destination permissions. On Linux, mv always preserves the original permissions even across filesystems. To force inheritance on Linux you must do it manually.
🏢
On Windows — moving to a different building means following that building's rules. On Linux — your personal rulebook travels with you regardless of which building you move into.
⊞ Windows NTFS
Cross-volume move = copy + delete. A brand-new file is created at the destination. It inherits the destination folder's ACL permissions. The original is then deleted.
🐧 Linux
mv always preserves original permissions regardless of which filesystem the destination is on. To force inheritance, use chmod --reference manually.
Visual Simulation
VOLUME 1
💿
HR /
drwxr-xr-x
📄 Report.docx rw-r--r--
→
mv (cross)
VOLUME 2
💾
/tmp/Different_Volume/
drwxrwxrwx
📄 Report.docx rw-r--r-- ✓ Linux
bash — solomon@PSBOS-SSeifu
$ mkdir /tmp/Different_Volume && chmod 777 /tmp/Different_Volume
$ mv HR/Report.docx /tmp/Different_Volume/
$ ls -l /tmp/Different_Volume/Report.docx
-rw-r--r-- 1 solomon solomon 0 May 4 09:00 Report.docx
✓ LINUX: Permissions preserved despite cross-volume move

# To manually force destination permissions:
$ chmod --reference=/tmp/Different_Volume /tmp/Different_Volume/Report.docx
-rwxrwxrwx 1 solomon solomon 0 May 4 09:00 Report.docx
→ NOW matches folder — but only because we forced it manually
Terminal — Try It Yourself
# Create different volume simulation
$ mkdir /tmp/Different_Volume
$ chmod 777 /tmp/Different_Volume
# Move across volumes
$ mv HR/Report.docx /tmp/Different_Volume/
# Verify — Linux preserves original permissions
$ ls -l /tmp/Different_Volume/Report.docx
-rw-r--r-- 1 solomon solomon 0 May 4 09:00 Report.docx
# Force destination folder permissions manually:
$ chmod --reference=/tmp/Different_Volume /tmp/Different_Volume/Report.docx
💡
Key Insight: chmod --reference=source target reads the permission bits from source and applies them to target. This is the Linux way to manually simulate Windows NTFS inheritance behavior. Without it, permissions always travel unchanged with the file.

Scenario 04 / 04
Copying Data to a Different Volume
Copying to a different volume always creates a brand-new file. On Windows the copy inherits the destination folder's permissions. On Linux, the original permissions are preserved (filtered by umask) — exactly the same behavior as copying within the same volume. The destination folder has no automatic effect.
🗂️
Whether you copy to the same building or a different building — on Linux the copy always carries the original rulebook (filtered through umask). On Windows the copy always follows the destination building's rules, no matter where it comes from.
⊞ Windows NTFS
A brand-new file is created on the other drive. It inherits the destination folder's permissions — same behavior as copying within the same volume. Source permissions are not carried over.
🐧 Linux
cp to a different volume preserves original permissions (filtered by umask). Destination folder has no effect. Use chmod --reference to force it.

# Verify — permissions preserved on Linux

-rw-r--r-- 1 solomon solomon 0 May 4 09:00 Report_copy2.docx

💡
Bottom Line: On Linux it does not matter whether you use mv or cp, and it does not matter whether the destination is the same volume or a different one — permissions always travel with the file. Destination folder permission inheritance is a Windows NTFS concept. On Linux you must apply it manually using chmod --reference.
