Transport Layer — TCP, UDP & Port Numbers

The Transport Layer is Layer 4 of the OSI model. It acts as the traffic manager for data — ensuring the right data gets to the right application on the right device, efficiently and reliably.

What the Transport Layer Does
Right Data, Right Application

An IP address gets the packet to the correct device. The Transport Layer ensures it reaches the correct application on that device — browser, email client, FTP, VoIP.

Reliable or Fast Delivery

The Transport Layer decides how data is sent. With guaranteed delivery when accuracy matters (TCP), or with speed when timing matters more (UDP).

Manages Data Flow

It ensures the receiver is not overwhelmed and the sender is not bottlenecked. Data flows smoothly between sender and receiver, even across different networks.

Simple Analogy
Think of it like sending multiple packages through a delivery service. Each package needs a destination address (IP address) AND a specific department to reach (port number). The Transport Layer ensures every package reaches the correct department without getting lost or mixed up.
TCP vs. UDP — The Core Comparison
TCP — Transmission Control Protocol

Reliable, ordered delivery. TCP ensures data arrives correctly, in order, and without errors.

  • Breaks data into segments and assigns each a sequence number
  • Receiver sends an acknowledgement for every segment received
  • If no acknowledgement arrives within the time limit, TCP retransmits only the missing segment — not the entire message
  • Reassembles segments in the correct order at the destination
  • Slower than UDP — reliability has a cost
Use TCP for: Web browsing (HTTP/HTTPS), email (SMTP, IMAP, POP3), file transfers (FTP) — any application where accuracy is critical.
UDP — User Datagram Protocol

Best effort delivery — fast but no guarantees. UDP sends data without ensuring delivery, ordering, or error checking.

  • No acknowledgements — sender does not wait for confirmation
  • No retransmission — if a packet is lost, it is gone
  • No ordering — packets may arrive out of sequence
  • Significantly faster than TCP — no overhead
  • Packet loss is usually unnoticeable in real-time applications
Use UDP for: Live streaming, VoIP calls, online gaming, DNS lookups — any application where speed matters more than perfection.

Simple Analogy
TCP is like certified mail — you get a receipt, the recipient signs for it, and if it gets lost the postal service sends another one. UDP is like dropping a postcard in the mailbox — fast, no confirmation, and if it gets lost nobody sends another one.
Port Numbers — What They Are and Why They Matter
What is a Port Number?

A port number is a numeric identifier in every TCP and UDP segment. It directs incoming data to the correct application or service on a device.

Every packet carries two port values:

  • Source port — assigned by the client OS (dynamic/ephemeral) — acts as the return address
  • Destination port — the well-known port of the requested service on the server
The Apartment Building Analogy

IP address = Building address — identifies which device (computer, phone, server) on the network.

Port number = Apartment number — identifies which specific application or service on that device should receive the data.

Without the port number, data arrives at the right building but nobody knows which door to deliver it to.
Three Port Number Categories — Managed by ICANN
Well-Known Ports — 1 to 1023

Used for standard common services — HTTP, HTTPS, DNS, FTP, SSH, SMTP. Strictly controlled by ICANN.

These are the port numbers every cybersecurity professional must memorize. They appear directly on the Security+ exam.

Registered Ports — 1024 to 49151

Used for specific applications registered by organizations. Instant messaging applications and many third-party software products use ports in this range.

Officially assigned by ICANN to avoid conflicts between different software applications.

Private/Dynamic Ports — 49152 to 65535

Also called ephemeral ports. Automatically assigned by the OS as source ports for outgoing connections. Exist only for the duration of one conversation.

Released back to the pool when the connection closes.

Sockets and Socket Pairs
What is a Socket?

A socket is the unique combination of an IP address and a port number.

Example: 192.168.1.5:52847

  • IP address: 192.168.1.5
  • Port number: 52847

A socket uniquely identifies one endpoint of a network connection.

What is a Socket Pair?

A socket pair is the combination of the client socket and the server socket — four values together that uniquely identify one complete network conversation.

  • Client socket: 192.168.1.5:52847
  • Server socket: 192.168.1.7:80

No two simultaneous connections on the internet can have the same socket pair.

Why Socket Pairs Keep Conversations Separate

A PC with one IP address can maintain dozens of simultaneous connections to the same server — each conversation gets a different dynamic source port, creating a unique socket pair for each one.

  • FTP connection: 192.168.1.5:1305 → 192.168.1.7:21
  • Web connection: 192.168.1.5:1099 → 192.168.1.7:80
  • Second web tab: 192.168.1.5:2203 → 192.168.1.7:80

Same IP address — different source ports — completely separate conversations. The OS uses the socket pair to match every incoming response to the correct application.

IP Address + Port Number = Socket
Client Socket + Server Socket = Socket Pair → uniquely identifies one complete network connection
The netstat Command
What netstat Does

netstat stands for Network Statistics. It is a command-line tool that displays all active TCP connections and their status on a device.

It shows:

  • Proto — TCP or UDP
  • Local Address — device IP and source port
  • Foreign Address — remote IP and destination port
  • State — ESTABLISHED, LISTENING, TIME_WAIT
netstat -n — The Security Professional's Version

By default, netstat resolves IP addresses to domain names and port numbers to service names — which takes time and can be misleading if DNS is compromised.

The -n flag skips DNS resolution:

  • Shows raw IP addresses instead of domain names
  • Shows numeric port numbers instead of service names
  • Faster output — no DNS lookup delay
  • Tamper-resistant — bypasses poisoned DNS records
Why netstat Matters for Cybersecurity

Unexplained TCP connections indicate a potential security threat. Unauthorized access may occur if unknown entities establish connections to the local host.

  • Every connection on a device should have a known and expected reason for existing
  • Malware almost always establishes a network connection — either to receive commands or send stolen data
  • An unexpected connection to an unfamiliar IP on an unusual port is a strong indicator of compromise
  • Common malware port: 4444 — a device with an active connection on port 4444 to an unknown IP needs immediate investigation
  • In incident response, netstat -n is one of the first commands a security analyst runs on a suspected compromised device
Connection States: ESTABLISHED = data actively flowing. LISTENING = service waiting for incoming connections. TIME_WAIT = connection in the process of closing cleanly.
How Everything Works Together — One Complete Picture
1

Application Layer — Request Generated

You type https://example.com. The browser generates an HTTP request and the Application Layer assigns destination port 443 (HTTPS).

2

Transport Layer — TCP or UDP Selected

Transport Layer selects TCP (reliable delivery for web traffic). A dynamic source port (e.g. 52,847) is assigned. The segment now has source port 52,847 and destination port 443.

3

Internet Layer — IP Addresses Added

Source IP (your device) and destination IP (the web server) are added. The packet is ready to be routed across the internet.

4

Network Access Layer — Frame Built and Sent

MAC addresses are added for local delivery. The frame travels through switches and routers hop by hop — MAC addresses rebuild at each hop, IP and port stay the same throughout.

5

Server Receives — Port Directs to Correct Service

Server receives the packet, checks destination port 443, hands it to the HTTPS web server application. Response comes back to your source port 52,847 — your browser receives it, not your email client.
